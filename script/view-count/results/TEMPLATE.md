# 게시글 조회수 증가 V1~V3 성능 측정 결과

> 사용법: 이 파일을 `results/YYYY-MM-DD-<설명>.md` 로 복사한 뒤 값을 채운다.
> 예) `cp results/TEMPLATE.md results/2026-07-13-hotrecord.md`

## 측정 환경

| 항목 | 값 |
|------|-----|
| 측정 날짜 | YYYY-MM-DD |
| Git 커밋 | `<git rev-parse --short HEAD>` |
| 앱 인프라 사양 | ECS on EC2 t3.small ×N / (또는 로컬 <CPU/RAM>) |
| DB 인프라 사양 | MySQL EC2 t3.small / (또는 로컬) |

| innodb_buffer_pool_size | <예: 1G> |

| 워밍업 여부 | <예: 동일 조건 1회 선행 실행 후 본 측정> |
| k6 버전 | v0.5x |
| POST_ID (fixed) | 6000000 |
| POST_ID_MIN ~ MAX (profile) | 5000001 ~ 6000000 |

## 표 1 — fixed 모드 (단일 핫 레코드 경합)

`POST_MODE=fixed` 로 게시글 하나에 쓰기를 집중시켜 락 전략별 경합 내성을 비교.
RATE 를 단계적으로 올리며 버전만 교체해 동일 조건으로 측정한다.

### RATE = 100 req/s

| 버전 | 실측 RPS | avg(ms) | p50 | p95 | p99 | max | 에러율 | 정합성* |
|------|----------|---------|-----|-----|-----|-----|--------|---------|
| V1 (낙관) |     |         |     |     |     |     |        |         |
| V2 (비관) |     |         |     |     |     |     |        |         |
| V3 (원자) |     |         |     |     |     |     |        |         |

### RATE = 300 req/s

| 버전 | 실측 RPS | avg(ms) | p50 | p95 | p99 | max | 에러율 | 정합성* |
|------|----------|---------|-----|-----|-----|-----|--------|---------|
| V1 (낙관) |     |         |     |     |     |     |        |         |

| 버전 | 실측 RPS | avg(ms) | p50 | p95 | p99 | max | 에러율 | 정합성* |
|------|----------|---------|-----|-----|-----|-----|--------|---------|
| V1 (낙관) |     |         |     |     |     |     |        |         |
| V2 (비관) |     |         |     |     |     |     |        |         |
| V3 (원자) |     |         |     |     |     |     |        |         |

- \* 정합성: `verify-integrity.sql` 전/후 델타 == k6 성공 요청 수 이면 O.
- RATE: constant-arrival-rate 목표 도착률. 실측 RPS 는 k6 `http_reqs` 기준.
- V2 는 락 대기가 커넥션 풀을 잠식하면 실측 RPS 가 RATE 에 못 미칠 수 있다 — 그것 자체가 결과.

## 표 2 — profile 모드 (현실 트래픽 분포)

`POST_MODE=profile` — 최신글 집중(45%/30%) + 검색 유입(15%) + 롱테일(10%) 분포로
게시글을 샘플. 쓰기가 분산될 때 세 버전의 차이가 얼마나 줄어드는지 확인한다.

| 버전 | RATE(req/s) | 실측 RPS | avg(ms) | p50 | p95 | p99 | max | 에러율 |
|------|-------------|----------|---------|-----|-----|-----|-----|--------|
| V1 (낙관) |        |          |         |     |     |     |     |        |
| V2 (비관) |        |          |         |     |     |     |     |        |
| V3 (원자) |        |          |         |     |     |     |     |        |

- 롱테일 세그먼트(depth 5001+)는 버퍼 풀 미스로 디스크 I/O 가 얹힐 수 있다 —
  fixed 모드보다 p99 가 오히려 커지는 구간이 있는지 관찰.

## 표 3 — V1 낙관적 락 재시도 통계

경합이 심할수록 재시도가 늘고(회당 +10ms 대기), 10회 소진 시 500 으로 실패한다.

| RATE(req/s) | 재시도 소진율(500)* | p95 에서 10ms 계단 관찰 | version 증가량** | 성공 수와 일치 |
|-------------|---------------------|-------------------------|------------------|----------------|
| 100         |                     |                         |                  |                |
| 300         |                     |                         |                  |                |
| 500         |                     |                         |                  |                |

- \* k6 `view_count_retry_exhausted` (V2/V3 는 항상 0 이어야 정상).
- \*\* `verify-integrity.sql` 의 version 델타. 총 커밋 횟수 = view_count 델타와 같아야 한다.

## 표 4 — 락 관찰 스냅샷 (부하 중 lock-waits.sql)

같은 RATE 의 fixed 모드 부하 중 캡처한 값. 전/후 델타는 [4]절 누적 카운터 기준.

|------|----------------------------------|-----------------------------|-------------------------------|------------------------|
| V1 (낙관) |                             |                             |                               |                        |
| V2 (비관) |                             |                             |                               |                        |
| V3 (원자) |                             |                             |                               |                        |

- 예상: V2 에서 대기 행/델타가 크고, V3 는 거의 0, V1 은 대기 대신 재시도(표 3)로 나타난다.

## 표 5 — EXPLAIN 요약

`explain/v*-*.sql` 각 쿼리의 핵심 지표. 세 버전 모두 PK 단건 접근으로 동일해야 하며,
